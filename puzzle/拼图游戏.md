# 拼图游戏 — 制作思路

## 一、项目概述

一个运行在浏览器中的网格吸附式拼图游戏。图片被均分为 M×N 个矩形切片，打乱后玩家通过拖拽将切片归位。相邻切片在位置正确时会自动"拼合"成一个大切片，后续拖拽按整体移动。所有切片归位即游戏完成。

**技术栈**：纯 HTML5 Canvas + 原生 JavaScript，无任何框架或外部依赖，单文件即可运行。

**设计风格**：浅色背景（`#f2f2f7`），白色卡片式画布容器，圆角描边切片，蓝色组轮廓。

## 二、布局设计

### 移动端 + 桌面端统一布局

采用自上而下的三段式布局，手机和桌面共用同一套 HTML 结构：

```
┌──────────────────────────────┐
│  🧩 拼图游戏       [− 4×3 +] │  ← 顶部工具栏
├──────────────────────────────┤
│                              │
│     ┌────────────────┐      │
│     │                │      │
│     │    Canvas      │      │  ← 游戏主区域 (flex:1 自动撑满)
│     │    游戏主区域   │      │
│     │                │      │
│     └────────────────┘      │
│                              │
├──────────────────────────────┤
│  [👁 显示原图] [📷 换图片] [🔄 重新打乱]  │  ← 底部工具栏
└──────────────────────────────┘
```

- **顶部工具栏**：标题 + 网格切片数量切换（− / N×M / +），最大宽度 800px
- **游戏主区域**：`flex: 1` 自动占满剩余空间，Canvas 居中适配
- **底部工具栏**：预览原图、换图片、重新打乱三个按钮，水平居中
- Canvas 容器最大宽度 800px，高度自动适配可用空间（不超过宽度 × 0.85）
- 四周 margin 自适应（`max(24px, min(cw,ch) × 0.06)`），小屏不浪费空间
- 图片等比缩放填满网格区域
- 按钮使用 `border-radius: 20px+` 胶囊形 + 大触屏热区（≥34px），方便手指操作

### 移动端特殊适配

- `overflow: hidden` 防止整页滚动
- `-webkit-tap-highlight-color: transparent` 消除 iOS 点击高亮
- `touch-action: manipulation` 消除 300ms 点击延迟
- `user-select: none` 防止拖拽时选中文字
- 按钮最小高度 34px，符合手指触控推荐尺寸
- `max-width: 400px` 媒体查询下进一步缩小间距和字号

## 三、核心数据结构

### 3.1 切片元数据（不可变）

```js
pieces[id] = {
  id,                    // 0 ~ TOTAL-1
  correctRow,            // 正确位置的行号
  correctCol,            // 正确位置的列号
}
```

每个切片有唯一的 `id`，以及它在完整图片中"应该"处于的行列位置。这两个值在初始化时确定，之后永不改变，作为判定归属的"身份证"。

### 3.2 网格状态（可变）

```js
grid[r][c] = pieceId     // 第 r 行第 c 列当前放置的是哪个切片
```

`grid` 是一个 ROWS×COLS 的二维数组，记录每个格子上当前放的切片 id。这是游戏的核心状态——所有移动操作本质上就是修改 `grid` 中 pieceId 的位置。

### 3.3 并查集（Union-Find）

```js
parent[id] = parentId    // 并查集父节点数组
```

用于追踪哪些切片"连接在一起"成为一个组。两个切片如果同时在网格中相邻、且相对位置与它们的正确位置关系一致，则它们属于同一组。

并查集提供三个操作：
- **`ufFind(x)`**：查找 x 的根节点（带路径压缩）
- **`ufUnion(a, b)`**：合并 a 和 b 所在的组
- **`getGroup(pid)`**：获取 pid 所在组的所有成员 id

### 3.4 拖拽状态

```js
isDragging         // 是否正在拖拽
dragPieceId        // 被拖拽的"锚点"切片 id
dragGroupIds       // 被拖拽的整组成员 id 列表
dragStartX/Y       // 鼠标/手指按下时的坐标
dragAnchorCellX/Y  // 锚点格子左上角的 canvas 坐标
dragDx/Dy          // 当前拖拽位移量（相对按下位置）
```

## 四、游戏初始化流程

```
loadImage()
  │
  └─→ image.onload
        │
        ├─→ layout()        计算网格布局参数
        ├─→ initGame()      创建切片 + 洗牌
        └─→ render()        首次绘制
```

### 4.1 布局计算 `layout()`

1. 根据游戏区域可用空间确定 Canvas 尺寸（最大 800px，高度不超过宽度 × 0.85）
2. 四周留自适应 margin：`max(24px, min(cw,ch) × 0.06)`
3. 按图片宽高比在可用区域内居中缩放，得到 `gridW × gridH`
4. 计算每个 cell 的尺寸：`cellW = gridW / cols`，`cellH = gridH / rows`

### 4.2 洗牌算法 `shuffleGrid()`

采用 **Fisher-Yates（Knuth）洗牌算法** 对整个 grid 数组进行随机排列：

```
1. 将 grid[r][c] 展平为一维数组 flat[0..TOTAL-1]
2. 从末尾向前遍历，每个位置与前方随机位置交换
3. 将打乱后的一维数组填回 grid[r][c]
4. 检查是否恰好完成 → 若是则重新洗牌
```

关键点：切片始终吸附在网格格子内，不会随机散落在任意坐标。这简化了交互模型——玩家只需关心"哪个切片去哪格"，不需要微调位置。

### 4.3 初始连接建立 `rebuildAllConnections()`

洗牌完成后，遍历所有相邻格子对，调用 `canConnect()` 判断是否可以拼合，能拼合则 `ufUnion()` 合并。这样游戏开始时就已经可能有部分切片恰好相邻且方向正确，形成一个初始组。

## 五、连接判定算法

### 5.1 核心函数 `canConnect(pidA, pidB)`

判断两个切片是否可以"拼合"为一个组，需同时满足两个条件：

**条件一：当前网格位置相邻**

```js
dr = posB.row - posA.row    // 必须是 -1, 0, 或 1
dc = posB.col - posA.col
|dr| + |dc| === 1           // 曼哈顿距离为 1（四邻域相邻）
```

**条件二：相对方向一致**

```js
correctDr = pieceB.correctRow - pieceA.correctRow
correctDc = pieceB.correctCol - pieceA.correctCol

dr === correctDr  &&  dc === correctDc
```

即：B 在 A 的右侧（当前位置），且 B 的正确位置也在 A 的正确位置的右侧。这意味着这两个切片不仅在空间中相邻，而且它们的"合影关系"是正确的。

### 5.2 组的概念

通过并查集，所有互相 `canConnect` 的切片被聚合为一个"组"。一个组代表一组已经正确拼合在一起的切片。视觉上：
- 组内切片之间无间隙，形成无缝画面
- 组的外边缘绘制蓝色发光轮廓线（沿实际形状走线，非矩形包围盒）
- 拖拽组内任意切片时，整个组一起移动

## 六、移动与交换逻辑

### 6.1 整体流程

```
pointerdown → 识别点击的切片，获取其组(groupIds)
pointermove → 更新 dragDx/DragDy，实时重绘
pointerup   → 计算目标 cell，执行 moveGroup()
```

### 6.2 `moveGroup(anchorId, targetRow, targetCol)` 详解

这是整个游戏最核心的算法，实现"整组移动到目标位置，同时交换被挤出的切片"。算法分五步：

**第一步：计算位移量**

```js
anchorPos = findGridPos(anchorId)       // 锚点当前 grid 坐标
dRow = targetRow - anchorPos.row        // 要移动的行偏移
dCol = targetCol - anchorPos.col        // 要移动的列偏移
```

**第二步：计算组内所有切片的新旧位置**

```js
groupCells = [{pid, r, c, dr, dc}, ...]  // 旧位置（含相对锚点偏移）
newCells   = [{pid, r: r+dRow, c: c+dCol}, ...]  // 新位置
```

边界检查：任一新位置超出 `[0, ROWS)` × `[0, COLS)` 则移动失败。

**第三步：收集被挤出的切片**

遍历所有新位置，找出当前占有者且不在本组内的切片：

```
displaced = [{pid, fromR, fromC}, ...]
```

同时收集源位置中不会被新位置覆盖的"空位"：

```
freeSlots = [{r, c}, ...]  // 旧格子中、不被新位置重叠的部分
```

**第四步：执行三步交换**

```
1. 清空旧位置（非重叠部分标记为 -1）
2. 将 displaced 填充到 freeSlots（一一对应）
3. 将组放置到新位置
```

**第五步：清理残余 + 重建连接**

```js
// 如果 displaced 多于 freeSlots，多余的 displaced 放回剩余 -1 位置
for (r, c in grid) {
  if (grid[r][c] === -1 && hasMoreDisplaced) {
    grid[r][c] = displaced[next].pid
  }
}
rebuildAllConnections()
```

### 6.3 交换示意图

以 2×2 组向右移动 1 格为例：

```
移动前：              移动后：
┌───┬───┬───┐        ┌───┬───┬───┐
│ A │ B │ C │        │ C │ A │ B │
├───┼───┼───┤   →    ├───┼───┼───┤
│ D │ E │ F │        │ F │ D │ E │
└───┴───┴───┘        └───┴───┴───┘

组 = {A, B, D, E}，向右移动 1 格
displaced = {C, F}  （被挤出）
freeSlots = 左列两格  （组移出后空出）
结果：C→左列上格，F→左列下格
```

## 七、渲染管线

渲染采用**三遍绘制**策略，确保正确的层叠关系：

### 第一遍：绘制所有非拖拽切片

```js
for (r, c) in grid:
  if pid 正在被拖拽 → 跳过
  if pid 所属组大小 ≥ 2 → 无间隙绘制（组内无缝）
  else → 带 2px 间隙 + 白色圆角描边
```

### 第二遍：绘制非拖拽组的轮廓

沿每个组（大小 ≥ 2）的**实际外边界**绘制蓝色轮廓线，而非简单矩形包围盒。

轮廓算法：
1. 构建组的 cellSet：`Set<"r,c">`
2. 收集外部水平边：当 `(r,c)∈组` 但 `(r-1,c)∉组` → 这是顶部外边
3. 同理收集底边、左边、右边
4. 合并连续边段，逐段绘制直线（带 1.5px 间距）

### 第三遍：绘制拖拽中的切片和轮廓

拖拽中的切片叠加在最上层，跟随手指位移 `(dragDx, dragDy)` 偏移：
- 组内无间隙
- 使用稍粗的蓝色轮廓（3px vs 2.5px）

### 单片渲染细节

```
独立切片（未拼合）：
  ┌────────────────────┐
  │  ← gap 2px →      │
  │  圆角矩形裁剪      │  cornerR = 4px
  │  白色描边 + 阴影   │
  │  drawImage()       │
  └────────────────────┘

组内切片（已拼合）：
  ┌────────────────────┐
  │  矩形裁剪（无圆角） │
  │  无间隙            │
  │  drawImage()       │
  │  轮廓由外部统一绘制 │
  └────────────────────┘
```

### 完成判定

渲染时检查 `isComplete()`：遍历所有格子，判断每个 piece 的 `correctRow/Col` 是否与当前 `grid` 位置一致。全部正确则在画面中央叠加半透明白色背景 + 绿色"🎉 恭喜完成！"文字。

## 八、交互设计

### 8.1 Pointer Events 统一模型

使用 Pointer Events API（`pointerdown/move/up/cancel`）统一处理鼠标和触屏，无需分别绑定 mouse/touch 事件。Pointer Capture 确保手指滑出 Canvas 区域后仍能跟踪拖拽。

### 8.2 点击判定

```js
cellAt(cx, cy):
  col = floor((cx - gridX) / cellW)
  row = floor((cy - gridY) / cellH)
  return {row, col} 或 null（超出网格）
```

### 8.3 拖拽位移

拖拽时不直接修改 grid，而是记录偏移量 `(dragDx, dragDy)`，渲染时叠加到切片位置：

```js
实际绘制位置 = cellCanvasX(col) + dragDx  // 拖拽中
实际绘制位置 = cellCanvasX(col)           // 静止时
```

好处：拖拽过程中 grid 保持不变，释放时一次性计算目标位置并更新 grid，避免中间状态污染。

### 8.4 目标判定

释放时计算锚点切片的中心点落在哪个 cell：
```js
锚点中心 X = dragAnchorCellX + dragDx + cellW / 2
锚点中心 Y = dragAnchorCellY + dragDy + cellH / 2
targetCell = cellAt(锚点中心X, 锚点中心Y)
```

### 8.5 拼合反馈

释放后比较移动前后的组成员数量：
```js
oldGroupSize = dragGroupIds.length
moveGroup(...)
newGroup = getGroup(dragPieceId)
if (newGroup.length > oldGroupSize) → playSnapSound()
```

有新连接形成时播放"咔嗒"提示音，给予即时反馈。

### 8.6 预览模式

点击"显示原图"按钮或按 `P` 键，画布切换为预览模式：
- 绘制半透明黑色遮罩
- 显示完整原始图片 + 白色网格线
- 显示"👁 预览模式 | 点击任意位置关闭"提示
- 点击画布任意位置或再次按按钮退出预览

### 8.7 完成弹窗

拼图完成时：
1. 画布内显示"🎉 恭喜完成！"文字（立即）
2. 500ms 后弹出 HTML 弹窗（`popIn` 缩放动画），提供两个按钮：
   - **🔄 再来一局**：当前难度重新打乱
   - **🔥 挑战更难**：增加切片数后重新开始
3. 点击弹窗遮罩区域也可重新开始

### 8.8 键盘快捷键（桌面端）

| 按键 | 功能 |
|------|------|
| `P` | 切换预览原图 |
| `R` | 重新打乱（不刷新页面） |

### 8.9 移动端触屏优化

- `touch-action: none` 在 Canvas 上阻止浏览器默认手势（滚动、缩放）
- `touchstart` 事件 `preventDefault()` 阻止双指缩放手势
- 按钮使用 `touch-action: manipulation` 消除 iOS 300ms 点击延迟
- `-webkit-tap-highlight-color: transparent` 消除点击蓝色高亮
- `user-select: none` / `-webkit-touch-callout: none` 阻止长按菜单

## 九、音效设计

使用 **Web Audio API** 合成音效，无需加载外部音频文件：

```
OscillatorNode (sine wave)
  频率: 1800Hz → 800Hz (指数衰减, 60ms)
  音量: 0.35 → 0.001 (指数衰减, 80ms)
```

生成短促清脆的"咔嗒"声，模拟拼图卡入到位的感觉。`AudioContext` 懒加载，仅在首次播放时创建。

## 十、响应式设计

### Canvas 尺寸自适应

```js
resizeCanvas():
  w = min(容器宽度, 800px)
  h = w × 0.82
  canvas.width = w, canvas.height = h
```

- 监听 `window.resize`，200ms 防抖后重新布局+渲染
- CSS `canvas { width: 100%; height: auto }` 保证 CSS 尺寸与 canvas 像素尺寸一致
- `touch-action: none` 阻止浏览器默认手势
- `user-select: none` 阻止文本选择
